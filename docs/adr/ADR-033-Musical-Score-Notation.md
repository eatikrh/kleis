# ADR-033: Musical Score Notation via LilyPond

**Status:** Proposed  
**Date:** 2026-03-14  
**Related:** ADR-012 (Document Authoring), ADR-023 (Kleist Template Externalization), ADR-018 (Universal Formalism)

---

## Context

Kleis generates academic papers by compiling `ArxivPaper` data structures into Typst
source, which Typst then renders to PDF. The pipeline is:

```
ArxivPaper -> compile_arxiv_paper -> .typ -> typst compile -> .pdf
```

This pattern — structured data, declarative compilation, external rendering — is
domain-agnostic. Music notation is a natural next domain: a musical score is a
structured object with well-defined syntax, semantics, and rendering rules. The
same pipeline applies:

```
Score -> compile_score -> .ly -> lilypond -> .pdf
```

### Why Music Fits the Kleis Philosophy

From the `.cursorrules` Level 3 philosophy:

> "Any domain that has notation, rules, and outputs fits the same pattern.
>  Kleis provides the substrate."

| Component      | Kleis Feature         | Music Domain                              |
|----------------|-----------------------|-------------------------------------------|
| **Notation**   | `.kleis` data types   | Pitch, duration, measure, staff, score    |
| **Rules**      | `axiom` in structures | Measure completeness, voice-leading, counterpoint |
| **Verification** | Z3 solver backend  | Well-formedness, constraint satisfaction  |
| **Output**     | LilyPond rendering    | Engraved sheet music (PDF)                |

### Why LilyPond

Typst lacks a mature music notation package. The `staves` package (v0.1.0) handles
single staves with individual notes — not complete scores. `typst-kyphu` wraps
`abc2svg` for ABC notation but ABC is limited for complex music.

LilyPond is the gold standard for automated music engraving:

- Open source (GPL), maintained since 1996
- Handles multi-voice, multi-staff, dynamics, articulations, lyrics, repeats
- Programmatic: source is a text format, generated by code
- Produces publication-quality PDF output

The relationship is exactly analogous to Typst for papers: Kleis owns the
data model and compilation; LilyPond owns the rendering.

---

## Decision

**Create a `sheet_music.kleis` template in `stdlib/templates/` that defines
musical notation data types and compiles them to LilyPond source.**

### Architecture: Five-Layer Theory

A musical score conflates five distinct concerns. This ADR separates them:

```
┌────────────────────────────────────────────────────────────────────┐
│ Layer 1: SYNTAX — symbolic tokens and local musical objects        │
│   Pitch, Duration, Accidental, Rest, Chord, Clef, KeySig, TimeSig │
└────────────────────────────────────────────────────────────────────┘
                              ↓
┌────────────────────────────────────────────────────────────────────┐
│ Layer 2: STRUCTURE — hierarchical and temporal organization        │
│   Measure, Voice, Staff, Part, Section, Score                     │
└────────────────────────────────────────────────────────────────────┘
                              ↓
┌────────────────────────────────────────────────────────────────────┐
│ Layer 3: SEMANTICS — musical meaning and interpretive constraints  │
│   Staccato = shortened duration, Slur = phrase continuity,        │
│   enharmonic identity (G♯ ≠ A♭ semantically)                     │
└────────────────────────────────────────────────────────────────────┘
                              ↓
┌────────────────────────────────────────────────────────────────────┐
│ Layer 4: PERFORMANCE — executable realization constraints          │
│   Timing, dynamics, pedal, ornament execution, tempo rubato       │
└────────────────────────────────────────────────────────────────────┘
                              ↓
┌────────────────────────────────────────────────────────────────────┐
│ Layer 5: RENDERING — graphical realization (engraving) constraints │
│   Staff layout, spacing, stem direction, beam grouping,           │
│   collision avoidance, glyph placement                            │
└────────────────────────────────────────────────────────────────────┘
```

**Key insight:** A score is the invariant object from which multiple views
(sheet music, MIDI performance, analytical reduction) can be derived under
constraints. This is the same projection structure as POT:

- Sheet music = a rendering projection
- MIDI = a performance projection
- Harmonic analysis = a semantic projection

Each projection loses information; the fiber contains what is lost.

### Phase 1 (This ADR): Layers 1-2 + LilyPond Rendering

Phase 1 defines data types for syntax and structure (Layers 1-2), compiles
them to LilyPond, and produces sheet music. This gets working output
immediately.

**Data types:**

```kleis
data NoteName = C | D | E | F | G | A | B
data Accidental = Natural | Sharp | Flat
data Pitch = P(NoteName, Accidental, ℤ)
data Duration = Whole | Half | Quarter | Eighth | Sixteenth | Dotted(Duration)
data Event = Note(Pitch, Duration) | Rest(Duration) | Chord(List, Duration)
data Clef = Treble | Bass | Alto | Tenor
data KeySig = KeySig(NoteName, String)
data TimeSig = TimeSig(ℤ, ℤ)
data Measure = Measure(List)
data StaffContent = Notes(Clef, KeySig, TimeSig, List)
data ScoreMeta = ScoreMeta(String, String, String)
data Score = Score(ScoreMeta, List)
```

**Compiler functions:**

| Function | Output |
|----------|--------|
| `compile_pitch(p)` | `"c''"`, `"fis'"`, `"bes,"` |
| `compile_duration(d)` | `"4"`, `"8."`, `"1"` |
| `compile_event(e)` | `"c''4"`, `"r8"`, `"<c' e' g'>4"` |
| `compile_measure(m)` | Space-separated events |
| `compile_staff(s)` | `\new Staff { \clef ... \key ... }` |
| `compile_score(s)` | Full LilyPond document |

**Basic verification (example blocks):**

- Measure duration sums to time signature
- Pitch within instrument range
- Valid time signature (denominator is power of 2)

### Phase 2 (Future): Axiomatic Engraving

Formalize LilyPond's engraving heuristics as Kleis axioms:

- Beam grouping rules (meter-dependent)
- Stem direction (voice/register-based)
- Spacing proportional to rhythmic density
- Collision avoidance between glyphs
- Accidental placement legibility

Instead of heuristic C++ code deciding layout, Z3-verified constraints
produce provably correct engravings. The unique contribution: ask
"does this rendering satisfy all engraving rules?" and get a
machine-checked answer.

### Phase 3 (Future): Native Typst Rendering

Once engraving rules are formalized (Phase 2), build a Typst rendering
backend that matches LilyPond quality. This eliminates the external
dependency:

```
Score -> compile_score_typst -> .typ -> typst compile -> .pdf
```

The formalized constraints from Phase 2 drive layout decisions.
The entire pipeline lives in one ecosystem.

### Semantic Layer (Future): Counterpoint and Harmony

Layers 3-4 enable formal verification of musical properties:

```kleis
structure Counterpoint {
    axiom no_parallel_fifths : ...
    axiom no_parallel_octaves : ...
    axiom stepwise_resolution : ...
}

structure HarmonicAnalysis {
    operation chord_function : Chord -> ChordFunction
    axiom cadence_resolution : ...
}
```

This goes beyond notation into music theory verification — "does this
passage satisfy the rules of four-voice counterpoint?" checked by Z3.

## Planned Refinements (Post Phase 1)

These architectural upgrades are noted for future phases. Phase 1 intentionally
uses a simpler model that maps directly to LilyPond.

### Time-Based Model

Phase 1 uses `Measure(List)` — a container of events. This is LilyPond-centric.
The deeper abstraction is time-continuous:

```kleis
data TimeSpan = Span(ℚ, ℚ)   // start, duration (rational for exact subdivision)

data MusicalObject =
    NoteObj(Pitch, TimeSpan)
  | RestObj(TimeSpan)
  | ChordObj(List, TimeSpan)
```

With this model, measures become constraints, not containers. Voices become
ordered sets over time. This enables tuplets, cross-measure ties, and
polyrhythm without special cases.

### Control Flow

Notation has control flow: repeats, endings, jumps (D.C. al Fine, D.S. al Coda).
Phase 1 uses flat measure lists. The full model needs:

```kleis
data Section =
    Linear(List)
  | Repeat(Section)
  | RepeatWithEndings(Section, List)
  | Jump(String)
```

With a traversal function: `play_order : Score -> List` that expands control
flow into a linear event sequence (what MIDI needs).

### Canonical Score

Define normalization axioms for what constitutes a well-formed score:

```kleis
structure CanonicalScore {
    axiom consistent_durations : ...
    axiom no_overlapping_events : ...
    axiom normalized_structure : ...
}
```

### Projection Functions and Compatibility

```kleis
operation render  : Score -> Rendering
operation perform : Score -> Performance
operation analyze : Score -> SemanticView

// The key verification questions:
// satisfies(performance, score)     — does this MIDI realize this score?
// valid_rendering(rendering, score) — does this engraving respect this score?
// equivalent(score1, score2)        — are these the same piece?
```

The score is coordinate-free; LilyPond and MIDI are different coordinate
systems for the same invariant object.

## Consequences

**Positive:**
- Demonstrates Kleis as a universal notation substrate beyond mathematics
- Reuses the existing template/compilation infrastructure
- Sheet music output with no Rust changes
- Opens the path to formal music theory verification

**Negative:**
- External dependency on LilyPond (mitigated by Phase 3 plan)
- LilyPond compilation is slower than Typst
- Music notation is a deep domain; Phase 1 covers basics only

**Neutral:**
- The five-layer decomposition parallels POT's projection-fiber structure,
  suggesting a deeper theoretical connection between the two domains
